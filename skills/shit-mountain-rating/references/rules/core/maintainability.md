# 可维护性评分规则

分析代码可维护性。

## 检测项

- 修改影响范围（改一个逻辑需要改几个文件）
- 可扩展性（加一个新功能需要改多少处）
- 模块独立性（一个模块是否可以独立理解和替换）
- 硬编码配置（魔法数字、写死的 URL/路径/凭据，非集中管理）
- 缺乏抽象（if-else 链、switch 蔓延、类型判断代替多态）
- 模块边界模糊（业务逻辑泄漏到 UI 层、数据访问泄漏到 Service 层）

## 阈值定义

| 指标 | 警告 | 严重 |
|------|------|------|
| 修改影响范围 | 涉及 3-5 文件 | >5 文件 |
| 硬编码配置 | 少量魔法数字 | 密钥/URL 硬编码 |
| 缺乏抽象 | 2 处同类模式 | 3 处以上同类模式 |
| 模块边界 | 偶有越界 | 系统性越界 |
| 可扩展性不足 | 加功能需改 3-5 处 | >5 处 |

## 评分公式

```
分数 = 100 - (修改影响严重 × 10) - (硬编码严重 × 10) - (缺乏抽象严重 × 8) - (模块边界严重 × 8) - (可扩展性严重 × 8) - (警告 × 3)
```

最低 0 分。

## 检测方式

- 分析 git log 中文件的 change coupling（同时变更频次）
- 扫描硬编码数字/URL/密钥模式
- 识别 if-else 链和 switch 中的类型判断
- 检测跨层引用
