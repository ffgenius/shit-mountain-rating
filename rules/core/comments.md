# 注释标记评分规则

分析代码中的标记注释，统计数量和分布。

## 检测项

- `TODO`（待完成的功能）
- `FIXME`（已知需要修复的 bug）
- `XXX`（需要关注的代码，通常有隐患）
- `HACK`（临时方案/变通实现）
- `@deprecated`（已废弃但仍在使用的接口）
- 被注释掉的代码块（>10 行连续注释代码）

## 阈值定义

| 指标 | 警告 | 严重 |
|------|------|------|
| FIXME 数量 | 3-10 个 | >10 个 |
| TODO 数量 | 10-20 个 | >20 个 |
| HACK 数量 | 3-5 个 | >5 个 |
| XXX 数量 | 3-5 个 | >5 个 |
| 废弃仍用 | 1-2 个 | >2 个 |
| 注释代码块 | 1-3 处 | >3 处 |

## 输出要求

不只统计总量，还要输出：
- 每个标记的文件、行号、首次提交时间（通过 git blame）
- 存活超过 6 个月的 FIXME/HACK 特别标注为"化石级"
- 标记密度（标记数 / 千行代码）

## 评分公式

```
分数 = 100 - (FIXME严重 × 10) - (HACK严重 × 5) - (XXX严重 × 4) - (TODO严重 × 2) - (废弃仍用严重 × 5) - (注释代码块严重 × 4) - (化石级标记 × 5) - (警告 × 1)
```

最低 0 分。

## 检测方式

- 正则扫描注释中的标记关键词
- git blame 获取每个标记的首次提交时间
- 计算标记密度和趋势
