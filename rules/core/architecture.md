# 架构评分规则

分析项目架构质量。

## 检测项

- 分层架构是否清晰
- 模块职责是否单一
- God Object（全能对象，>300 行或 >15 个公有方法）
- God Function（全能函数，>50 行或 >3 个独立职责）
- Feature Coupling（功能耦合，不相关功能挤在同一模块）
- Layer Violation（层违规，底层依赖上层）
- 同级循环依赖（同层模块互相引用）

## 阈值定义

| 指标 | 警告线 | 严重线 |
|------|--------|--------|
| God Object | >300 行或 >15 方法 | >500 行或 >20 方法 |
| God Function | >50 行或 >3 职责 | >100 行或 >5 职责 |
| Layer Violation | 偶发 1-2 处 | 系统性 >2 处 |
| Feature Coupling | 2 个不相关功能共存 | 3 个以上不相关功能共存 |
| 模块职责 | 职责界限模糊 | 完全无职责划分 |
| 同级循环依赖 | — | 任意数量即严重 |

> 注：God Object / God Function 的行数阈值与复杂度规则共用一套标准。架构维度从"职责分配"角度评估，复杂度维度从"理解和测试难度"角度评估，两者互补但不重复主要扣分。

## 评分公式

```
分数 = 100 - (God Object 严重 × 15) - (God Function 严重 × 10) - (Layer Violation 严重 × 10) - (Feature Coupling 严重 × 5) - (职责不清模块 严重 × 5) - (同级循环依赖 × 10)
```

最低 0 分。

## 检测方式

- 扫描类/文件行数和公有方法数
- 分析 import/export 依赖方向
- 检查底层模块是否引用了上层符号
- 识别同层模块间的双向依赖
